HR1システム全体の戦略レビューを実施してください。
「HR管理 + 社内ポータル」統合プラットフォームとして、MVP1リリースの品質評価と今後の成長戦略を策定します。

## 前提条件

- **現在のフェーズ**: MVP1リリース前
- **方針**: リーガル要件（年末調整、e-Gov連携、給与計算等）は当面対象外。自社完結で提供できる機能を優先的に充足させる
- **ターゲット**: 従業員10〜200名の中小企業（専任人事がいない/兼務）

## 手順

5つのサブエージェントを**並行**で起動し、以下を調査してください：

### エージェント1: hr1-console (Next.js管理コンソール)
- `hr1-console/src/app/` 以下の全ルート・ページを確認
- 各機能の実装完成度（TODO/FIXME、未実装スタブ、プレースホルダー）
- テストカバレッジ（`__tests__/` + `e2e/` ディレクトリ）
- パフォーマンス（仮想スクロール、キャッシュ戦略、リアルタイム更新）
- ビルド・テスト実行: `npm run build && npm run test`

### エージェント2: hr1-applicant-app (Flutter応募者アプリ)
- `lib/features/` 以下の全機能を確認
- 各画面の実装完成度（TODO/FIXME、未実装スタブ）
- クリーンアーキテクチャの遵守度（Screen/Controller/Repository分離）
- CLAUDE.mdのFlutter UIガイドライン違反の残存数（lightTextSecondary, onSurface.withValues, ConsumerStatefulWidget, Center(child:Text('エラー')), 相対import, FilledButton）
- ビルド検証: `flutter analyze`

### エージェント3: hr1-employee-app (Flutter社員アプリ)
- エージェント2と同じ観点で調査
- 社内ポータル機能の完成度（ポータルホーム、社員名簿、お知らせ、Wiki、グループチャット）
- ビルド検証: `flutter analyze`

### エージェント4: Supabaseバックエンド + hr1_shared
- `supabase/migrations/` のスキーマ整合性、マイグレーション数
- RLSポリシーの網羅性と安全性
  - helper関数（get_my_role, get_my_organization_ids, get_my_accessible_thread_ids）の一貫性
  - 旧ポリシー（直接JOIN版）の残存確認
  - **RLS変更の影響チェーン**: テーブル間の参照関係に再帰リスクがないか（`pg_policies`のqualカラムで逆引き検索）
- Edge Functionsの実装品質
- `hr1_shared/` の共有コードの品質

### エージェント5: サービス成長戦略
コードは読まず、**事業・プロダクト**の観点で分析する。リーガル要件（年末調整、e-Gov、給与計算）は対象外とし、自社完結の機能に集中する。

- **現在の機能で初期顧客を獲得できるか？** 足りないものは何か？
- **獲得と定着の戦略**: 何で顧客を獲得し、何で使い続けてもらうか？
- **MVPとして不足している基本機能**: メール通知、CSV一括インポート、無料プラン等
- **既存機能の磨き込み優先度**: どの機能を深くすべきか、どの機能は現状で十分か？
- **価格戦略**: SMBに刺さる価格帯と無料プランの範囲
- **Go-to-Market**: 営業チームなしでSMBにリーチする方法
- **リテンション**: DAU/MAU向上のための施策。ポータル戦略は機能するか？
- **ロードマップの妥当性**: STRATEGIC_REVIEW.mdの現行ロードマップを評価し、自社完結機能に絞った現実的な優先順位を提示

## レポート形式

`STRATEGIC_REVIEW.md` を以下の構成で**上書き更新**してください。

**重要**: 対応済みの項目（実装完了・修正完了・ビルド確認済み等）は全て削除すること。「やるべきこと」「ブラッシュアップ」「コード品質」には**未対応のもののみ**を記載する。前回のレポートに残っていても、今回の調査で対応済みと判明した項目は必ず削除する。

### プロダクトビジョン
ターゲット、競合との違い、獲得戦略と定着戦略

### 1. やるべきこと
🔴MVP1リリースに必須 / 🟡MVP2（次フェーズ） / 🟢将来 に分類。
各項目に**カテゴリ**（ポータル/HR/事業/品質）、**根拠**、**工数**（S/M/L/XL）を付与。
リーガル要件（年末調整、e-Gov、給与計算）は「将来」に分類し、自社完結機能を優先すること。

### 2. 既存機能のブラッシュアップ
機能はあるが改善が必要なもの。重要度と対象を付与。

### 3. コード品質
技術的負債。重要度・規模・対象を付与。

### 4. 現状の強み
競合との差別化ポイント。

### 5. 競合比較
機能マトリクス。

### 6. ロードマップ
自社完結機能に集中した現実的なフェーズ分け。
