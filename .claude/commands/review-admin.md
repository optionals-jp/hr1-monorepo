hr1-admin（Next.js HR1運営管理画面）のコードレビューを実施してください。

## 概要

hr1-admin は HR1 サービス運営者向けの管理画面です。契約企業管理、プラン管理、MRR ダッシュボード等を提供します。
hr1-console（テナント企業向け管理コンソール）とは別のアプリです。

## 手順

1. `git diff HEAD -- hr1-admin/` と `git diff origin/main...HEAD -- hr1-admin/` で変更を確認する
2. 変更がある場合は変更されたファイルを全て読み込む。変更がない場合は `hr1-admin/src/` 配下の全ページ・コンポーネント・hooks を対象にアプリ全体レビューを実施する
3. 以下の観点でレビューする

## レビュー観点

### 必須チェック
- **バグ・ロジックエラー**: off-by-one、null参照、型不一致、条件分岐の漏れ
- **セキュリティ**: 運営管理画面のため認証/認可が特に重要。一般ユーザーやテナント管理者がアクセスできないこと（`profiles.role = 'platform_admin'` 等の検証がサーバー側で行われているか）。XSS、SQLインジェクション、機密情報のハードコード
- **データ整合性**: 契約・プラン・課金データの整合性。誤操作防止（確認ダイアログ等）
- **HTML ネスト制約**: `<button>` 内に `<button>` をネストしていないか（hydration エラー防止）

### 暫定的な対応・ワークアラウンドの禁止（🔴 要修正）

**基本方針**: 暫定対応・ワークアラウンド・応急処置は一切禁止。根本原因を特定して正しく修正する。大規模改修になっても構わないので、**安全で可読性の高いコード**を常に優先する。

- **インライン再定義**: import が削除された enum / 定数を関数内やファイル冒頭にインライン再定義していないか → 正しい `lib/constants/` に配置する
- **TODO/FIXME/HACK コメント**: `// TODO: 後で` / `// 暫定` / `// 一時的` 等の先送りコメントを新規追加していないか
- **`any` / `as unknown as` / `@ts-ignore` / `@ts-expect-error` の濫用**: 型エラーを握り潰していないか
- **try/catch の握り潰し**: `catch (e) {}` / `catch (e) { return null }` でエラーを黙殺していないか
- **ハードコード分岐での応急処置**: 特定IDや特定ロールで `if` 分岐し本来のバグを回避していないか
- **コピペで類似関数を新規作成**: 既存関数を修正するのを避け、類似関数を増やしていないか → 既存を拡張する
- **`XxxV2` / `XxxNew` / `XxxLegacy` 命名での新旧並存**: 置き換え完了まで責任を持って完遂する
- **フィーチャーフラグの放置**: 切替完了後の旧フロー削除をサボっていないか
- **エラー戻り値スワップ**: エラー時に `null` を返して呼び出し元に「なかったことにする」ていないか → `{data, error}` 形式か throw

**指摘フォーマット**: `[ファイル:行] の記述は暫定対応。根本原因: [原因]。正しい修正: [大規模でも構わないので正しい方針]`

### クリーンアーキテクチャ

hr1-console と同一のレイヤー構成（Presentation → Hooks → Repository → Supabase）に従う。

- ❌ **Components が Supabase を直接呼んでいないか**: 必ず hooks 経由
- ❌ **Hooks が UI ライブラリに依存していないか**: hooks 内で `useToast` / `showToast` を呼ばない。JSX を返さない
- ❌ **Hooks が Supabase を直接呼んでいないか**: Repository 経由で DB アクセス
- ❌ **lib/ が UI ライブラリに依存していないか**
- ✅ **Components → Hooks → Repository → Supabase** の依存方向

### 共有UIパッケージ（@hr1/shared-ui）
- `@hr1/shared-ui` のコンポーネント（Button, Card, Dialog 等）を優先使用。ローカルで再実装していないか
- hr1-console と重複するコンポーネントがあれば `@hr1/shared-ui` への移行を提案する

### コード品質
- **命名**: 変数・関数名が処理内容を適切に表しているか
- **重複**: 同じロジックがコピペされていないか
- **未使用コード**: importやvariableが使われずに残っていないか
- **エラーハンドリング**: 適切にエラーが処理されているか（過剰なハンドリングは不要）

### Next.js / React 固有
- **"use client" / "use server"**: クライアント・サーバーコンポーネントの分離が適切か
- **React Hooks**: useEffect の依存配列、SWR の使い方
- **状態管理**: 不要な再レンダリングを引き起こしていないか

### Supabase クライアント
- **SSR クライアント**: `@supabase/ssr` 経由で使用しているか
- **権限**: 運営管理者のみがアクセスできるテーブル・RPC のアクセス制御
- **型定義**: `src/types/database.ts` との整合性

### UI / スタイリング
- **UIテキスト**: 日本語が適切か（運営者向けの表現になっているか）
- **shadcn/ui**: 共通コンポーネント（Card, Table, Badge, Dialog 等）を適切に使用しているか
- **Tailwind CSS**: 任意値クラスではなく標準クラスを優先しているか

### 運営管理固有
- **ダッシュボード**: KPI（MRR、契約数、社員数）の計算が正しいか
- **契約管理**: ステータス遷移（trial → active → suspended → cancelled）が正しいか
- **プラン管理**: 料金計算の整合性
- **企業管理**: テナントデータの参照が正しいか（cross-tenant でのデータ漏洩がないか）

### ビルド・Lint チェック
- レビュー完了後、`cd hr1-admin && npm run format && npm run lint && npm run test && npm run build` を実行し、エラーがないか確認する

## 出力フォーマット

レビュー結果を以下の形式で報告してください：

### 🔴 要修正（マージ前に修正が必要）
### 🟡 推奨（修正を推奨するが必須ではない）
### 🟢 良い点（特に良い実装）

各項目には **ファイル名:行番号** と **理由** を明記してください。

問題がない場合は「レビュー完了：問題なし ✅」と報告してください。
