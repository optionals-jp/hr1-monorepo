Push前のコードレビューを実施してください。

## 手順

1. `git diff` でステージング済みの変更を確認し、未ステージの変更も `git diff HEAD` で確認する
2. 変更されたファイルを全て読み込む
3. 以下の観点でレビューする

## レビュー観点

### 必須チェック
- **バグ・ロジックエラー**: off-by-one、null参照、型不一致、条件分岐の漏れ
- **セキュリティ**: XSS、SQLインジェクション、認証/認可の漏れ、機密情報のハードコード
- **データ整合性**: DB操作の整合性、外部キー制約、トランザクション境界

### 暫定的な対応・ワークアラウンドの禁止（🔴 要修正）

**基本方針**: 暫定対応・ワークアラウンド・応急処置は一切禁止。問題を見つけたら**根本原因を特定して正しく修正する**。大規模改修になっても構わないので、**安全で可読性の高いコード**を常に優先する。

#### ワークアラウンドの典型パターン（検出したら 🔴 要修正）

- **インライン再定義**: import が削除された enum / 定数 / 型を関数内やファイル冒頭にインライン再定義している（例: `const StepStatus = { ... } as const` をファイル途中に書く）→ 正しい constants モジュールに再配置する
- **TODO/FIXME/HACK コメントでの先送り**: `// TODO: 後で直す` / `// FIXME` / `// HACK` / `// 暫定対応` / `// 一時的に` 等のコメントを新規追加している → チケット化して即修正する
- **`any` / `as unknown as` / `@ts-ignore` / `@ts-expect-error` の濫用**: 型エラーを握り潰している → 正しい型定義に修正する
- **try/catch による握り潰し**: `catch (e) {}` や `catch (e) { return null }` でエラーを黙殺している → 根本原因を特定し、適切にハンドリングまたは上位に伝播させる
- **条件分岐での応急処置**: `if (特定のID === "...")` のようなハードコード分岐で本来のバグを回避している → データ設計 / ロジック設計を見直す
- **コピペによる局所修正**: 既存関数を修正するのを避けて、類似関数を新規作成しコピペで差分対応 → 既存関数を正しく修正するか共通化する
- **削除すべきコードのコメントアウト**: `// 古い実装` `// 旧コード` のようなコメントアウトを残している → 削除する（git 履歴に残る）
- **既存実装と矛盾する設計の並行追加**: 既存の Repository / Hook / コンポーネントと目的は同じだが設計が異なる新規実装を並行で追加している → 既存実装を拡張するか、既存実装を置き換える
- **命名での先送り**: `XxxV2` / `XxxNew` / `XxxLegacy` のような名前で新旧並存させている → 置き換え完了まで責任を持って完遂する
- **環境変数やフラグでの分岐放置**: `if (process.env.NEW_FLOW)` のようなフィーチャーフラグを立てたまま旧フローを残している → 切替が終わったら旧フローを削除する
- **エラーの戻り値スワップ**: エラー時に `null` / `undefined` / 空配列を返して呼び出し元に「なかったことにする」 → 呼び出し元が区別できるよう `{data, error}` 形式で返すか throw する

#### 大規模改修を恐れない

- 根本修正の副作用で**複数ファイルの修正が必要**になっても、それが正しいなら実施する
- 既存の API / 関数シグネチャを変えるのが正しいなら変える（後方互換のためのラッパーを作らない）
- 共通化すべきロジックが見つかったら、呼び出し元を全て置き換える（2箇所だけ置き換えて残りはワークアラウンドで逃げる、はしない）
- **「安全で可読性の高いコード」を「動くだけのコード」より優先する**
- レビュアーが「これはワークアラウンドでは？」と感じたら、その違和感を率直に指摘する

#### ワークアラウンドを指摘する際のフォーマット

```
[ファイル名:行番号] の [具体的な記述] は暫定対応。
根本原因: [原因の説明]
正しい修正: [大規模でも構わないので正しい修正方針を提示]
```

### コード品質
- **命名**: 変数・関数名が処理内容を適切に表しているか
- **未使用コード**: importやvariableが使われずに残っていないか
- **エラーハンドリング**: 適切にエラーが処理されているか（過剰なハンドリングは不要）

### 重複検出（最重要）

**基本方針**: 共通処理・共通コンポーネントを使用して重複を徹底的に避ける。既存の共通実装があるのに再実装している場合は 🔴 要修正。新規で書いたロジックが他箇所と類似している場合は共通化を提案する。

#### 手順
1. **変更箇所のロジックを抽出**: 追加/変更された関数・コンポーネント・ユーティリティを列挙する
2. **既存の共通実装を検索**: 以下の場所で類似実装がないか grep で必ず確認する
   - **Next.js 系**: `packages/shared-ui/src/`, `hr1-console/src/lib/`, `hr1-console/src/components/ui/`, 対象アプリの `src/lib/`, `src/components/`, `src/features/*/rules.ts`
   - **Flutter 系**: `hr1_shared/lib/src/widgets/`, `hr1_shared/lib/src/utils/`, `hr1_shared/lib/src/constants/`, 対象アプリの `lib/core/`, `lib/shared/`
3. **類似度の判定**: 関数名・シグネチャ・処理の骨格（3行以上の類似ロジック、同じ正規表現、同じバリデーションルール、同じ日付フォーマット等）を比較する
4. **類似が見つかった場合**: 既存実装を使うよう修正を提案する。既存実装が不十分な場合は既存側を拡張する方針を提案する（新規作成ではなく）

#### 重複の典型パターン（必ずチェック）

**Next.js 系**
- **UIコンポーネント再発明**: Button, Card, Dialog, Input, Badge, Table, Tabs 等を `@hr1/shared-ui` から import せずローカルで作っていないか
- **レイアウトパターン**: `sticky top-14 z-10` / `border-b-2 -mb-px` / テーブル白背景ラッパー等を各画面にインラインで書いていないか（`StickyFilterBar` / `TabBar` / `TableSection` を使う）
- **Supabase クエリ**: 同じテーブルへの類似クエリが複数の hooks から直接書かれていないか → Repository に集約
- **日付フォーマット / 通貨フォーマット / バリデーション**: `lib/datetime-utils.ts` 等の既存ユーティリティがあるのに独自実装していないか
- **定数・enum・ラベル**: ステータスラベル、ロール名等が画面ごとにハードコードされていないか → `lib/constants/` に集約
- **ビジネスルール**: フィルタリング・ソート・計算ロジックが複数コンポーネントに散在していないか → `features/*/rules.ts` に抽出
- **audit_logs 取得**: 各画面で個別に `audit_logs` を fetch していないか → `AuditLogPanel` の自動取得モードを使う
- **型定義**: 同一テーブル/RPC の型が複数ファイルで手書き定義されていないか → `src/types/database.ts` + 派生型を使う

**Flutter 系**
- **共通ウィジェット再発明**: `CommonButton`, `CommonCard`, `CommonDialog`, `CommonSheet`, `CommonSnackBar`, `EmptyState`, `ErrorState`, `LoadingIndicator`, `MenuRow`, `SearchBox`, `UserAvatar`, `OrgIcon`, `SkeletonBone` 等を `hr1_shared` から使わずローカルで作っていないか
- **デザイントークン**: `AppTextStyles`, `AppColors`, `AppSpacing` を使わず `TextStyle()` / `theme.colorScheme` / マジックナンバーを直接書いていないか
- **AsyncValue.when パターン**: `Scaffold → body → state.when(data/loading/error)` の構造を各画面で独自に書き直していないか
- **両アプリ共通ロジック**: hr1-applicant-app と hr1-employee-app に同じバリデーション/日付処理/サニタイズが存在していないか → `hr1_shared/lib/src/utils/` に集約

#### 判定ガイドライン

- **3回ルール**: 同じロジックが3箇所以上で使われている場合は必ず共通化する（2回目は許容、3回目で共通化）
- **3行以上の類似**: 3行以上の類似ロジックは共通化候補（1〜2行の糖衣構文は許容）
- **既存実装の拡張優先**: 共通化する際は既存の共通モジュール（`hr1_shared`, `@hr1/shared-ui`, `lib/` 配下のユーティリティ）を拡張する。並列で新しい共通モジュールを作らない
- **重複を指摘する際のフォーマット**:
  ```
  [ファイル名:行番号] で実装している処理は [既存モジュール/パス] と重複している。
  [既存実装を使う / 既存実装を拡張する] ことで重複を解消できる。
  ```
- ❌ **ワークアラウンドとしての重複実装は禁止**: 「既存実装に少し合わないから新しく書く」は不可。既存実装を拡張するか、既存実装側の設計を見直す

### プロジェクト固有
- **対象アプリの判別**: 変更ファイルのパスから対象アプリ（hr1-console / hr1-admin / hr1-employee-web / hr1-lp / Flutter 系）を把握し、各アプリのルールを適用する。より詳細なレビューは `/review-console`, `/review-admin`, `/review-employee-web`, `/review-applicant`, `/review-employee` を使用
- **UIテキスト**: 日本語が適切か（テナント管理者・運営者・社員・応募者のどの向けかで語彙を合わせる）
- **Supabaseクライアント（Next.js）**: `@supabase/ssr` 経由（`createBrowserClient` / `createServerClient`）で使用しているか
- **共有UI（Next.js）**: `@hr1/shared-ui` のコンポーネントをローカルで再実装していないか
- **型定義**: `src/types/database.ts` との整合性
- **テナント分離**: クエリに org_id フィルタが含まれているか。RLS との二重防御になっているか

## 出力フォーマット

レビュー結果を以下の形式で報告してください：

### 🔴 要修正（マージ前に修正が必要）
### 🟡 推奨（修正を推奨するが必須ではない）
### 🟢 良い点（特に良い実装）

各項目には **ファイル名:行番号** と **理由** を明記してください。

問題がない場合は「レビュー完了：問題なし ✅」と報告してください。
